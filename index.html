Chapter 1

The Bug

Every investigation starts with a report that looks impossible.

A system did something it should never do, and nobody can make it happen again on demand. That is the exact shape of the hardest bugs in distributed systems. They only appear under one unlucky ordering of events, and by the time anyone looks closely, the evidence is already gone. Traditional testing only checks the situations a person thought to imagine in advance, which means the scariest bugs, the ones nobody imagined, slip through completely.

Chapter 2

Package Your System

Antithesis cannot investigate a system it cannot run.

Before anything else happens, your whole system gets packaged as containers, wired together exactly the way it runs in production. This matters more than it sounds like it should, because the bugs that matter most rarely live inside one service alone. They live in the space between services, in the timing of one call reaching another, so testing a single service in isolation can never find them. Antithesis needs the full picture, not one piece of it.

Chapter 3

The Determinator

Antithesis does not guess. It captures reality.

Underneath everything Antithesis does sits a piece of technology it actually calls the Determinator, its own name for the deterministic hypervisor that runs your system. Real computers make thousands of tiny timing decisions every second, and none of them land quite the same way twice. The Determinator sits underneath your entire simulated system and makes every one of those decisions itself, recording exactly what happened instead of leaving it to chance.

the obstacle

The exact timing that caused a bug, like a confirmation arriving a few milliseconds late, is normally governed by real hardware. You cannot force real hardware to be unlucky the same way twice.

what is actually running underneath

This is not a modified operating system. Your unmodified Linux, your unmodified application code, all of it runs completely normally on top of the Determinator. The difference sits one level below that, in the hypervisor itself. Most CPU instructions execute at close to full host speed, because their results are already deterministic on real hardware, adding two numbers gives the same answer every time regardless of who is watching. The Determinator only has to step in for the instructions that are not naturally deterministic: reading the system clock, asking for a random number, checking how long an operation took. Those get trapped and handed a manufactured answer instead of a real one. That is the whole trick. It is not simulating a computer from scratch, it is running a real computer and quietly overriding the small number of places where real computers normally behave unpredictably.

Ready to record

Every dot above is one real event from the payment case, request, authorization, the delayed confirmation, the retry, and the duplicate write, recorded in the exact order it happened. Nothing here is guessed after the fact. It is captured live, in order, as it occurs.

Chapter 4

Same Seed, Same Bug

Because the Determinator controls every timing decision itself, an entire run can be captured as one number.

Give it the same seed twice, and it produces the identical run twice, down to the exact millisecond something went wrong. A bug found once is no longer a fluke. It becomes something you can call up on command.

applied to the payment case, try it yourself
Seed

Every seed always produces the same result, no matter how many times you run it. Type a different number and run it again. Some seeds turn up the bug, most will not, but whatever a given seed produces, it will produce that same thing every single time.

how this actually works

The seed is not a recording of what happened. It is the input to a generator that decides what happens. The Determinator, the hypervisor described in the last chapter, replaces every source of real world randomness on the machine, including thread scheduling, network delivery timing, and any random number your code calls, with its own pseudorandom generator. Give that generator the same seed twice, and it produces the identical sequence of decisions both times, the same way a well known formula always produces the same output for the same input. The bug is not saved anywhere. It is regenerated from the seed forward, which is why running the same seed twice gives you the exact same duplicate charge, down to the millisecond.

One more thing worth knowing: the seed that finds a real bug is almost never guessed at randomly and gotten lucky on the first try. Antithesis watches which lines of code and which branches get exercised as it runs, and when it notices something it has not seen before, a branch that almost never gets taken, a log line that says something should never happen, it treats that as a lead worth following and keeps exploring from there instead of starting over from scratch each time. That is what makes it possible to find a bug that depends on three separate unlikely things lining up at once, instead of needing to get equally lucky three times in a row.

Chapter 6

Rewind: Causality Analysis

Reproducing the bug is one thing. Knowing exactly where it broke is another.

Once a bug shows up, Antithesis can rewind the same timeline to different moments and explore forward again from each one, many times, with different fault patterns and different scheduling. It counts how often the bug still happens from each point in time. Put together, that produces a graph of bug probability climbing as the timeline gets closer to the failure, with a sharp jump exactly where the outcome became locked in.

Concretely, this works by walking backward through the exact decision points that were made along the way, the moments where the platform reached in and decided a random outcome, and testing a batch of different alternate outcomes from each one. If every alternate still leads to the same bug, that decision point was already too late, the bug was baked in before it. If some alternates avoid the bug, that decision point still mattered. Walking backward through enough of these points is what locates the exact jump on the graph, not a guess, a direct search.

reading the graph

Early in the timeline, the outcome could still go either way, so the probability is low. At the exact moment the delayed confirmation caused the retry logic to fire without checking the first attempt, the probability jumps hard. Rewind to any point after that jump, and the bug is already baked in.

Chapter 7

Properties, the Laws of the System

Teams define what correct means exactly once.

Instead of writing one handwritten test for one input, a property states a rule and Antithesis checks that rule against every single run it explores. An always property is a rule that must never break. A sometimes property is a rule that must actually be witnessed happening at least once, proof that the hard scenario was really tried and not just skipped.

the real obstacle, in Antithesis's own words

Even experienced engineers reach this point and ask, this sounds great, but what properties should my system have. That question is the actual bottleneck. It is not about understanding Antithesis. It is about understanding your own system well enough to state its rules out loud.

the method, how to find them in any system
1

Find your core guarantee, the one thing that must never be violated no matter what happens, such as conservation, uniqueness, or ordering. That becomes an always property.

2

Find what must actually be exercised to trust your resilience, such as a failover that really fires or a retry that really succeeds. That becomes a sometimes property.

3

State each one as a rule, not a scenario. Not test that node 3 crashes, but the system always recovers from any single node crashing.

applied to the payment case

The rule that made the causality graph in the last chapter possible at all was simple to state: a retried payment should never process twice. Antithesis did not need a thousand handwritten test cases to check that. It needed that one sentence, and it kept trying to break it until it found the one condition that did.

Chapter 8

The Pipeline

End to end, automated.

Every idea in this guide is one stage of the same automatic pipeline, and this is the actual order it runs in. You package your system as containers. You define a test template, the properties that state what correct means. The Determinator runs the whole thing over and over with different seeds, injecting faults along the way. The moment a property breaks, you get a Triage Report. Then, because everything was seeded, you replay the exact bug on demand.
