Chapter 1

The Bug

Every investigation starts with a report that looks impossible.

A system did something it should never do, and nobody can make it happen again on demand. That is the exact shape of the hardest bugs in distributed systems. They only appear under one unlucky ordering of events, and by the time anyone looks closely, the evidence is already gone. Traditional testing only checks the situations a person thought to imagine in advance, which means the scariest bugs, the ones nobody imagined, slip through completely.

Chapter 2

Package Your System

Antithesis cannot investigate a system it cannot run.

Before anything else happens, your whole system gets packaged as containers, wired together exactly the way it runs in production. This matters more than it sounds like it should, because the bugs that matter most rarely live inside one service alone. They live in the space between services, in the timing of one call reaching another, so testing a single service in isolation can never find them. Antithesis needs the full picture, not one piece of it.

Antithesis runs it in an isolated, repeatable environment
Same code. Same configuration. Same everything, every single time it runs.

Chapter 3

The Determinator

Antithesis does not guess. It captures reality.

Underneath everything Antithesis does sits a piece of technology it actually calls the Determinator, its own name for the deterministic hypervisor that runs your system. Real computers make thousands of tiny timing decisions every second, and none of them land quite the same way twice. The Determinator sits underneath your entire simulated system and makes every one of those decisions itself, recording exactly what happened instead of leaving it to chance.

the obstacle

The exact timing that caused a bug, like a confirmation arriving a few milliseconds late, is normally governed by real hardware. You cannot force real hardware to be unlucky the same way twice.

Ready to record

Every dot above is one real event from the payment case, request, authorization, the delayed confirmation, the retry, and the duplicate write, recorded in the exact order it happened. Nothing here is guessed after the fact. It is captured live, in order, as it occurs.

Chapter 4

Same Seed, Same Bug

Because the Determinator controls every timing decision itself, an entire run can be captured as one number.

Give it the same seed twice, and it produces the identical run twice, down to the exact millisecond something went wrong. A bug found once is no longer a fluke. It becomes something you can call up on command.

Seed

Every seed always produces the same result, no matter how many times you run it. Type a different number and run it again. Some seeds turn up the bug, most will not, but whatever a given seed produces, it will produce that same thing every single time.

Chapter 5

Fault Injection

Now Antithesis breaks things on purpose.

Most bugs do not appear when everything goes right. Antithesis finds them by deliberately making things go wrong: delaying a network call, crashing a service outright, duplicating a request, killing a process mid task, or corrupting a response before it arrives, all inside the safety of a simulation.

Chapter 6

Properties, the Laws of the System

Teams define what correct means exactly once.

Instead of writing one handwritten test for one input, a property states a rule and Antithesis checks that rule against every single run it explores. An always property is a rule that must never break. A sometimes property is a rule that must actually be witnessed happening at least once, proof that the hard scenario was really tried and not just skipped.

the real obstacle, in Antithesis's own words

Even experienced engineers reach this point and ask, this sounds great, but what properties should my system have. That question is the actual bottleneck. It is not about understanding Antithesis. It is about understanding your own system well enough to state its rules out loud.

the method, how to find them in any system
1

Find your core guarantee, the one thing that must never be violated no matter what happens, such as conservation, uniqueness, or ordering. That becomes an always property.

2

Find what must actually be exercised to trust your resilience, such as a failover that really fires or a retry that really succeeds. That becomes a sometimes property.

3

State each one as a rule, not a scenario. Not test that node 3 crashes, but the system always recovers from any single node crashing.

Chapter 7

The Pipeline

End to end, automated.

Every idea in this guide is one stage of the same automatic pipeline, and this is the actual order it runs in. You package your system as containers. You define a test template, the properties that state what correct means. The Determinator runs the whole thing over and over with different seeds, injecting faults along the way. The moment a property breaks, you get a Triage Report. Then, because everything was seeded, you replay the exact bug on demand.
